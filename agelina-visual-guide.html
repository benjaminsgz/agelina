Async orchestration framework

Agelina 的核心是一个把步骤编译成执行计划的异步编排内核。

这个项目最值得理解的不是 demo，而是“编排”这件事本身： 如何把步骤声明成元数据，如何在构建期验证依赖，如何在运行期推进 ready 节点， 以及如何把数据依赖和线程资源隔离放进同一个执行模型里。

      编排内核：Agelina 到底在“编排”什么

      编排不是业务逻辑本身，而是业务逻辑之间的执行关系。Agelina 把每个业务动作拆成 StepDefinition，再根据“顺序依赖”或“图依赖”生成执行计划。你可以把它理解成： 业务函数负责算什么，编排内核负责什么时候算、在哪里算、算完唤醒谁、失败时怎么收束。

      核心源码阅读顺序：StepDefinition -> AsyncStepFactory -> AsyncPipeline / SlotAsyncGraphBuilder -> SlotAsyncGraph。

      声明层

      把业务函数变成带名字和执行模式的步骤：name、mode、handler。

      计划层

      Pipeline 生成线性链；Graph 生成拓扑序、依赖表、slot 表和 terminal 出口。

      运行层

      每次执行创建独立运行帧，记录剩余依赖、slot 值、ready bit 和最终结果。

      调度层

      根据 ExecutionMode 路由到 IO、CPU 或当前线程，并把完成事件回传给编排器。

      编排器的输入不是“代码顺序”，而是“依赖关系”

      在 Pipeline 中，依赖关系天然等于添加顺序：第 N 步必须等第 N-1 步成功。 在 SlotAsyncGraph 中，依赖关系被显式拆成两类：

      关系 在哪里声明 编排器用它做什么
      控制依赖 dependencies 决定一个节点什么时候可以启动，编译为 remainingDependencies 和 dependentsByNode。
      数据依赖 readSlots / writeSlots 决定节点能读写哪些中间值，构建期验证读槽者必须依赖写槽者。
      资源依赖 ExecutionMode 决定步骤被路由到 IO pool、CPU pool，还是直接在调用线程执行。
      完成依赖 terminal 决定整个图什么时候对调用方完成，并保证 terminal 是图出口。

      两种编排模型的本质区别

      SlotAsyncGraph 的构建期编译过程

      1. 收集定义
      definitions 保存每个节点的名称、mode、dependencies、readSlots、writeSlots 和 evaluator。
      2. 解析符号槽
      SlotSymbolTable.Builder 把 "memberLevel" 这类名字分配为稳定 int slot id。
      3. 校验控制图
      检查缺失依赖、重复节点、环，并生成拓扑序。
      4. 校验数据图
      检查单写者原则，读 slot 的节点必须在控制依赖上依赖对应 writer。
      5. 编译运行结构
      生成 RuntimeNode[]、dependentsByNode、initialRemainingDependencies、terminalPath。

      SlotAsyncGraph 的运行期推进算法

      execute(context):
  frame = new GraphFrame(context)
  for each node:
    if initialRemainingDependencies[node] == 0:
      dispatchNode(node, frame)
  return frame.result

completeNode(node, error):
  if error:
    if node is on terminalPath:
      result.completeExceptionally(error)
    return

  if node is terminal:
    result.complete(terminalResult)

  for each dependent in dependentsByNode[node]:
    remaining = atomicDecrement(dependent)
    if remaining == 0:
      dispatchNode(dependent, frame)

      这个算法的重点是：Graph 不需要一个中央循环反复扫描所有节点。节点完成事件本身会推动下游依赖计数递减， 计数归零的瞬间就触发下一批节点。

      为什么要有 slot

      如果节点之间共享一个 Map，热路径会有哈希、装箱和并发访问成本。slot 把名字编译成数组下标，运行时直接读写。

      为什么要有 readyBits

      slot 数组保存值，readyBits 保存“这个值是否已经发布”。写 slot 后再 CAS 标记 ready，读节点先检查 ready。

      为什么要有 terminalPath

      图中可能存在与 terminal 无关的旁支。terminalPath 用于判断失败是否应该让整个执行结果失败。

      从源码看编排器的几个关键字段

      字段 / 类型 所在类 它解决的问题 理解方式
      List<AsyncStep<C>> AsyncPipelineBuilder 保存线性步骤，build 时按顺序组合。 链表式编排。
      Map<String, NodeDefinition> SlotAsyncGraphBuilder 保存用户声明的节点图。 构建期的源 AST。
      SlotSymbolTable SlotAsyncGraphBuilder / SlotAsyncGraph 把人类可读 slot 名转换成运行时 int id。 符号表，类似编译器。
      RuntimeNode[] SlotAsyncGraph 运行期不再查 Map，直接按数组索引访问节点。 编译后的执行计划。
      int[][] dependentsByNode SlotAsyncGraph 节点完成后快速知道要通知哪些下游节点。 反向邻接表。
      int[] remainingDependencies GraphFrame 每次执行独立记录每个节点还有多少上游没完成。 ready 调度计数器。
      Object[] slots SlotState 保存节点之间传递的中间值。 运行时数据平面。
      AtomicLongArray readyBits SlotState 跨线程发布 slot 就绪状态。 低成本可见性标记。

      AsyncPipeline：线性业务流的 fail-fast 链条

      Pipeline 适合登录、下单前置校验、按顺序补全上下文这类流程。它本质上把多个 AsyncStep 用 CompletableFuture.thenCompose() 串起来：上一步成功，下一步才启动；任何一步失败，后续跳过。

      auth demo 现在的链路是：loadUser -> verifyPassword -> issueToken -> LoginResponse。
      1. loadUser
      IO 线程池查询 MySQL，查不到用户抛出 BizException。
      2. verifyPassword
      CPU 线程池执行 BCrypt 校验，密码错误时 fail-fast。
      3. issueToken
      CPU 线程池调用 JwtUtils.createToken 生成 HS256 JWT。
      4. build response
      Controller 通过 WebFlux Mono.fromFuture 返回异步结果。
      return new AsyncPipelineBuilder<LoginContext>()
    .addStep(loadUser)
    .addStep(verifyPassword)
    .addStep(issueToken)
    .build();

// AsyncStep.then 的核心语义
ctx -> this.apply(ctx).thenCompose(next::apply)
      • 前置步骤失败，后续步骤不会运行。
      • 异常会沿着 CompletableFuture 链传播到 Controller。
      • demo 中由 GlobalExceptionHandler 把 BizException 转成 400 响应。
      • 如果线程池饱和且策略是 ABORT，调用方得到明确的拒绝异常。

      SlotAsyncGraph：并行 DAG 的编译期安全和运行期低开销

      DAG 适合报价预览这类“多个上游并行，最后汇总”的业务。Agelina 的特色是把符号槽名编译成整数 slot id， 运行时用 Object[] slots 和 AtomicLongArray readyBits 传递中间值。

      builder 会拒绝重复节点、缺失依赖、环、重复写 slot、读 slot 没有依赖路径，以及 terminal 后面继续挂下游节点。

      运行期帧

      每次 execute() 创建独立 GraphFrame：包含 slot 数组、readyBits、remainingDependencies 和 terminalResult。

      低开销数据通路

      节点完成时写 slots[id]，CAS 标记 ready bit，下游直接按 int id 读取，避免热路径 Map 查找。

      Spring 集成：starter 把执行基础设施放进容器

      threadpool-spring-boot-starter 依赖 autoconfigure 模块。引入 starter 后， Spring Boot 自动发布线程池、注册表、dispatcher、step factory、decorator 链和 @AsyncStepBean 扫描器。

      当前实现中，注解步骤会懒加载 AsyncStepFactory 并应用 CompositeStepDecorator。

      自动配置 Bean 流程

      ThreadPoolProperties
      绑定 threadpool.async.io/cpu 配置并做启动校验。
      ioExecutor / cpuExecutor
      通过 ThreadPoolFactory 创建命名线程池。
      ExecutorRegistry
      保存 ExecutionMode.IO/CPU 到 executor 的不可变映射。
      DefaultExecutionDispatcher
      实现普通 step 和 DAG node 的分发。
      AsyncStepFactory
      把 StepDefinition 转换成可执行 step。
      AsyncStepBeanPostProcessor
      扫描 @AsyncStepBean 方法并注册 stepDefinition.* 与 asyncStep.*。

      @AsyncStepBean 动态注册

      // 业务代码只写方法
@AsyncStepBean(name = "loadUser", mode = ExecutionMode.IO)
public LoginContext loadUser(LoginContext ctx) { ... }

// 容器启动后自动出现
stepDefinition.loadUser
asyncStep.loadUser

// asyncStep 会经过 CompositeStepDecorator
decorator.decorate("loadUser", step)

      这意味着业务方可以用代码式 builder，也可以通过注解把步骤发布为可注入 Bean。
