# なぜスレッドプールを分離するのか

バックエンドの業務処理は、もう単純な同期メソッドの呼び出しだけでは終わりません。  
1つのリクエストの中で、DBアクセス、外部API呼び出し、キャッシュ参照、認証計算、集計処理などが混在するのが普通です。

このとき、すべての処理を1つのスレッドプールに流し込むと、見た目はシンプルでも、実運用ではかなり早い段階で問題が出ます。

このプロジェクトでは、処理の性質に応じて実行先を分けるために、以下の 3 つの実行モードを用意しています。

- `IO`: DB、RPC、リモート呼び出しなどの待ち時間を含む処理
- `CPU`: 計算、変換、検証などのCPU集約型処理
- `DIRECT`: その場で即時実行してよい軽量処理

要するに、「非同期にする」ことが目的ではなく、「どの種類の仕事を、どの実行基盤に乗せるかを明示する」ことが目的です。

## 対象読者への前置き

この記事は、以下のような方を対象にしています。

- Spring Boot 環境で非同期処理を導入済み、または導入を検討中の方
- スレッドプールを使っているが、具体的なスレッド数設定に根拠がない方
- 高負荷時にレスポンス遅延が全域に波及する経験をされた方

後半のコード例は Java + Spring Boot を前提に記載しています。

# 1. なぜ隔離スレッドプールが必要なのか

## 単一スレッドプールの一番大きな問題

単一スレッドプールでは、速い処理と遅い処理、待ちが多い処理と計算が重い処理が、同じキューと同じワーカーを奪い合います。

たとえば次のような状況です。

- 外部API待ちで 100ms 止まる処理
- DBアクセスで 80ms 止まる処理
- パスワード検証や金額計算のようにCPUを使う処理
- 最終レスポンスを組み立てる軽量処理

これらを同じプールに載せると、IO待ちのタスクがスレッドを握ったままになり、CPUで即座に終わる仕事まで後ろに並ばされます。  
結果として起きるのは、単なる遅延ではなく「性質の違う仕事同士の巻き込み」です。

## 起きやすい障害

- IO待ちが増えると、CPUタスクが実行待ちになる
- CPUタスクが詰まると、リクエスト全体の完了が遅れる
- 軽量な終端処理までキュー渋滞に巻き込まれる
- 局所的な遅延がシステム全体のレイテンシ悪化に波及する

特に怖いのは、ボトルネックが1か所に集中したときに、関係ない処理まで一緒に死ぬことです。
本来は「外部依存が遅い」だけの問題が、「アプリ全体が遅い」に変わしてしまいます。

## 単一スレッドプールでは避けられない「不善循環」

単一プールには、もう一つ構造的な問題があります。それは**スレッド数の肥大化**を招きやすいことです。

IO待ち中含有が高いタスクが大量に入ると、パaranteeされたレイテンシを維持するためにスレッド数を増やincentiveが発生します。しかしスレッドが増えると、コンテキストスイッチのオーバーヘッドが増え、今度はCPU効率が落ちていきます。

```
スレッド数増加 → コンテキストスイッチ増加 → CPU効率低下 → さらにスレッド追加 ←←←
```

この不善循環は、プールを分割するだけで断ち切れます。IO は IO プールに任せて、CPU プールはコア数にあった控えめなサイズに保てるからです。

# 2. 分離すると何がうれしいのか

スレッドプールを `IO` と `CPU` に分離すると、少なくとも次のメリットがあります。

## 干渉を減らせる

IO待ちの増加が、そのままCPUタスクの遅延に直結しなくなります。  
逆にCPU負荷が上がっても、IOワーカーまで一緒に詰まりにくくなります。

## チューニングの単位が明確になる

単一プールだと「スレッド数を増やすべきか、キューを増やすべきか」の判断が曖昧です。  
分離していれば、

- IOプールは待ち時間を吸収するために大きめ
- CPUプールはコア数に合わせて抑制的

という基本方針を持てます。

## 障害時の影響範囲を限定できる

外部APIが遅くなったとき、影響をIOプール側に閉じ込めやすくなります。  
これは性能だけでなく、可用性の観点でも大きいです。

## 実行モデルをコードに表現できる

このプロジェクトでは、各ステップが `ExecutionMode` を宣言します。  
つまり「この処理はどこで走るべきか」が設計情報としてコードに残ります。

これは後から見たときにかなり効きます。  
非同期処理の保守がつらいのは、処理の流れよりも「どこで、何が、誰のスレッドで動くか」が見えなくなるからです。

# 3. 単一スレッドプールとの比較

比較すると、違いはかなりはっきりしています。

| 観点 | 単一スレッドプール | 分離スレッドプール |
| --- | --- | --- |
| タスクの干渉 | 起きやすい | 起きにくい |
| ボトルネックの特定 | 難しい | しやすい |
| チューニング | 全部まとめて調整 | IO/CPU別に調整 |
| 障害の波及 | 全体に広がりやすい | 局所化しやすい |
| 設計の明示性 | 低い | 高い |

単一プールの利点は、初期実装が単純なことです。  
ただし、トラフィックが増えたり、外部依存が増えたり、DAGのように並列分岐が入った瞬間に、シンプルさの利益が急速に薄れます。

逆に、分離プールは最初に少しだけ設計コストが増えます。  
それでも、後から性能改善と障害切り分けがしやすくなるので、実サービスでは回収しやすい投資です。

# 4. このプロジェクトでの実践

このリポジトリでは、処理を `AsyncPipeline` と `SlotAsyncGraph` の2つのモデルで表現しています。

- `AsyncPipeline`: 順次実行が中心のフロー向け
- `SlotAsyncGraph`: 依存関係を持つ並列実行向け

特に今回のテーマに相性が良いのは `SlotAsyncGraph` です。
依存関係のある複数タスクを並列に走らせつつ、実行先を `IO` / `CPU` / `DIRECT` で切り分けられるからです。

## ExecutionMode の実装例

`ExecutionMode` はスレッドプールの識別に使うシンプルな enum です。

```java
public enum ExecutionMode {
    /**
     * DB・RPC・外部APIなど、I/O待ちを含む処理。
     * 待ち時間を吸収するために比较大的なプールサイズにする。
     */
    IO,

    /**
     * 計算・変換・検証など、CPU集約型の処理。
     * コア数に合わせて抑制的にサイズを設定する。
     */
    CPU,

    /**
     * 即時実行してよい軽量処理。スレッドを占有させない。
     */
    DIRECT
}
```

各ステップでは、処理の内容に合わせてどのモードで実行するかを明示的に指定します。

```java
// 例: リクエスト検証は DIRECT（即時実行）
Slot.of("validateRequest", ExecutionMode.DIRECT, () -> validator.validate(request))

// 例: 在庫 조회は IO（外部依存を含む）
Slot.of("loadInventory", ExecutionMode.IO, () -> inventoryClient.findByProductId(productId))

// 例:  할인액 계산は CPU（CPU集約型）
Slot.of("calculateDiscount", ExecutionMode.CPU, () -> discountCalculator.compute(profile, items))
```

こうやって `ExecutionMode` を一つ一つ宣言することで、「この処理はどこで走るべきか」がコード上で明確になります。

## 例: 見積もりプレビュー処理

`/quotes/preview` のDAGでは、次のような構成を取っています。

- `validateRequest` は `DIRECT`
- `loadInventory` は `IO`
- `loadMemberProfile` は `IO`
- `loadProductAndBaseAmount` は `IO`
- `calculateMemberDiscount` は `CPU`
- `calculateCouponDiscount` は `CPU`
- `buildQuote` は `DIRECT`

ポイントは、3つのIO処理を並列に流し、その後の割引計算だけをCPUプールで処理していることです。

もしこれを単一プールで処理すると、

- IO待ちの3タスクが先にスレッドを専有する
- その完了後にCPU計算が順番待ちになる
- 負荷が高いと終端処理まで後ろに押し出される

という構造になりやすくなります。

一方、分離しておけば、

- IOはIOプールで並列に消化
- CPUはCPUプールで計算
- 軽量な終端処理は `DIRECT` で即時実行

という、かなり素直な流れになります。

## 実際の設定

今回の圧測では以下の設定を使っています。

```yaml
threadpool:
  async:
    io:
      core-size: 8
      max-size: 32
      queue-capacity: 16
      rejection-policy: CALLER_RUNS
    cpu:
      core-size: 4
      max-size: 4
      queue-capacity: 32
      rejection-policy: ABORT
```

この設定からも意図は明確です。

- IOは待ちを含むので拡張性を持たせる
- CPUはコア数に寄せて絞る
- 過負荷時はIO側で `CALLER_RUNS` によりバックプレッシャーをかける

## スレッド数の決め方

スレッド数の設定に迷ったときは、まずタスクの特性を分析してください。

| タスクの特性 | 推奨スレッド数 | 理由 |
| --- | --- | --- |
| CPU集約型 | `コア数 + 1` | コンテキストスイッチのオーバーヘッドを避ける |
| IO集約型 | `コア数 × 利用率 × (1 + 待時間/計算時間)` | Littleの法則に基づく算出 |
| 混合型 | 上記を分流 | 種類の違うタスクを混ぜない |

**Littleの法則**の適用例として、外部API呼び出し（待時間 200ms、計算時間 10ms）の場合：

```
スレッド数 = Nc × U × (1 + Ws/Cs)
           = 4 × 0.8 × (1 + 200/10)
           = 4 × 0.8 × 21
           ≈ 67
```

実際には、外部依存の同時接続数や下游サービスのキャパシティも考慮する必要があります。スレッド数を盲目的に増加させることがないようにしてください。

## キューサイズの設計思想

キューは「スレッドが処理を終えるまでの間にたまる仕事の缓冲地帯」です。ただし、無制限にキューを积むと、遅延が潜伏して問題の発見が遅れます。

今回の設定では、IOプールに `queue-capacity: 16` と小さめの値を採用し、`CALLER_RUNS` ポリシーで過負荷時のバックプレッシャーを明示的に発生させています。これは**遅延を隠さない**という設計判断です。

運用上 중요한のは、キューが満ちたときにアラートを上げられるようにすることです。

# 5. 圧測結果

今回のベンチマーク対象は `threadpool-dag-app` で、`wrk` による30秒計測を複数シナリオで行っています。

業務シナリオでは、IOステップとして次の遅延を模擬しています。

- `loadInventory`: 100ms
- `loadMemberProfile`: 90ms
- `loadProductAndBaseAmount`: 120ms

このため、単純直列なら理論上のIO合計は `310ms` です。  
一方、3つを並列化できれば理論上はおおよそ `220ms` 以下まで短縮できる見込みがあります。

## 結果サマリ

| シナリオ | 条件 | TPS | 平均レイテンシ |
| --- | --- | ---: | ---: |
| A | 1 thread / 2 connections | 16.38 | 121.64ms |
| A2 | 1 thread / 4 connections | 32.74 | 121.58ms |
| B | 2 threads / 8 connections | 64.70 | 123.30ms |
| C | 4 threads / 32 connections | 126.69 | 254.64ms |
| D | 8 threads / 64 connections | 138.83 | 460.14ms |
| E | 8 threads / 128 connections | 139.45 | 899.64ms |

## 読み取れること

### 低負荷時のレイテンシがかなり良い

軽負荷の A/A2/B では、平均レイテンシがほぼ `121ms - 123ms` に収まっています。  
これは直列合計 `310ms` よりかなり小さく、IO並列化が効いていることを示しています。

つまりこの構成では、単にスレッドを増やしたのではなく、

- 並列実行できる仕事をDAGで分解し
- IOプールに適切に載せ
- 終端を軽く保つ

ことで、レスポンス時間を圧縮できています。

### スループットは C から D あたりで頭打ち

TPSは A から C にかけてきれいに伸びますが、D/E では `約139 TPS` 前後でほぼ横ばいです。  
一方でレイテンシは `254ms -> 460ms -> 899ms` と急激に悪化しています。

これは典型的な「処理能力の上限に当たり、待ち行列だけが伸びている」状態です。

### ボトルネックはIOプール側

今回の構成では IO プールが `core-size: 8`、`max-size: 32`、`queue-capacity: 16` です。  
高負荷時にはこの領域が先に詰まり、`CALLER_RUNS` が発動して送信側に圧力が返っていると考えるのが自然です。

これは悪いことではありません。  
むしろ、無制限にキューを積んで遅延を隠すより、明示的に飽和させてシステムを守る方が運用上は安全です。

### 過負荷でも完全崩壊していない

最も高負荷の E でも、`4197` リクエスト中タイムアウトは `5` 件だけでした。  
TPSも `139.45` を維持しており、完全なスルーダウンには入っていません。

これは、隔離されたIOプールとバックプレッシャー設計が、過負荷時の崩れ方を比較的穏やかにしていると見られます。

# 単一スレッドプールだったらどうなるか

今回の実装は分離前後の直接比較ベンチではありませんが、単一プールにした場合に起きやすいことはかなり想像しやすいです。

- IO待ちタスクがCPUタスクの実行タイミングを押し下げる
- レイテンシの立ち上がりがもっと早い段階で始まる
- 高負荷時に終端処理まで巻き込まれて、タイムアウト率が上がりやすい
- ボトルネックが「プール全体の飽和」としてしか見えず、改善ポイントがぼやける

特にDAGのような「並列に投げる価値がある処理」が増えるほど、単一プールの欠点は拡大します。  
せっかく並列化しても、実行基盤が1つだと結局同じ待ち行列に戻ってしまうからです。

# まとめ

スレッドプールを分離する理由は、単に設計をきれいに見せるためではありません。  
処理の性質が違う仕事を同じ場所に押し込めないためです。

今回の実装と圧測から見えるポイントは次のとおりです。

- IOとCPUを分けることで、タスク同士の干渉を減らせる
- 実行モードを明示することで、設計と運用の両方が読みやすくなる
- DAGによる並列化と隔離プールの組み合わせは、レイテンシ短縮に効く
- 高負荷時はIO側が先に飽和するが、影響範囲をコントロールしやすい
- 単一プールより、性能改善の方向性が見えやすい

非同期処理で本当に難しいのは、「どう並列にするか」よりも「どの仕事を、どこに隔離して載せるか」です。  
その意味で、隔離スレッドプールは最適化テクニックではなく、実務的な安定運用の土台だと思っています。
