# 线程池物理隔离与容量评估架构

在现代高性能微服务中，CPU 利用率和系统响应时间在很大程度上取决于并发资源的调度与隔离机制。Agelina 秉承的设计哲学是：在多并发高吞吐系统中，**共享的通用线程池**是导致级联式性能雪崩的最主要根源。

本指南将深入剖析 Agelina 的核心物理资源隔离架构（IO 线程池、CPU 线程池与 DIRECT 模式直连），提供基于利特尔法则（Little's Law）的线程池容量计算数学模型，并指导如何在生产环境中微调这些配置参数。

---

## 共享线程池的致命缺陷

在典型的 Spring Boot 或 Java 并发开发中，开发人员通常会将所有异步任务提交给同一个共享的 `ForkJoinPool` 或标准的通用 `ThreadPoolExecutor`。当应用同时处理混合工作负载（既有快速的 CPU 计算，又有缓慢的 I/O 阻塞，如数据库查询、远程 HTTP 调用等）时，共享线程池就会产生**线程饥饿（Thread Starvation）**。

```
混合负载 -> 共享线程池 [-------------------]
           - 任务 A (I/O 密集型): 阻塞工作线程，等待外部 DB 响应 (耗时 200ms)
           - 任务 B (CPU 密集型): 快速解析一段 JSON (耗时 2ms)
           - 在高并发下，所有工作线程瞬间被任务 A 占满。
           - 任务 B 被积压在队列中，导致本来极快的 API 响应耗时暴增，甚至超时。
```

当共享线程池因阻塞式 I/O 任务被占满时，极快的 CPU 密集型任务被迫在阻塞队列中排队，导致系统响应延迟急剧升高，并最终引发全局超时雪崩。

## Agelina 线程池物理隔离机制

Agelina 强制执行物理级别的执行资源隔离。它将任务在运行时的执行模式分类并分配到三个完全隔离的执行通道中：

```
                  +----------------------------------------------+
                  |              ExecutionDispatcher             |
                  +----------------------------------------------+
                                         |
         +-------------------------------+-------------------------------+
         |                               |                               |
         v                               v                               v
+------------------+           +------------------+            +------------------+
|     IO 线程池     |           |    CPU 线程池     |            |   DIRECT 直连模式  |
| (AsyncIOExecutor)|           | (AsyncCPUExecutor|            | (调用者线程执行)  |
+------------------+           +------------------+            +------------------+
| - 阻塞式 I/O 操作 |           | - 纯计算密集型    |            | - 无锁零分配操作  |
| - 等待时间占比极高 |           | - 等待时间占比极低|            | - 零线程上下文   |
| - 动态扩缩容线程数 |           | - 严格绑定物理核心|            |   切换调度开销   |
+------------------+           +------------------+            +------------------+
```

### 1. IO 线程池 (`ExecutionMode.IO`)
* **设计定位**：专门用于处理包含阻塞行为的操作，如 SQL 数据库查询、HTTP/RPC 请求、外部缓存读写（Redis/Memcached）以及本地文件系统读写等。
* **运行特征**：线程配置数上限较高，采用可弹性收缩的空闲保留策略（Keep-Alive），并配合较大的阻塞等待队列。
* **核心目标**：安全地吸收并缓冲外部微服务或慢查询的耗时波动，防止其波及和拖慢纯 CPU 计算任务。

### 2. CPU 线程池 (`ExecutionMode.CPU`)
* **设计定位**：专门用于高频的纯计算型操作，如加解密运算、JSON/Protobuf 序列化与反序列化、业务逻辑校验及复杂数学计算。
* **运行特征**：线程数量严格按物理 CPU 核心数进行配额（Nc），队列容量设限极低，一旦过载立即触发中止型（Abort）拒绝处理。
* **核心目标**：最大程度减少操作系统（OS）的线程上下文切换开销，并保证 CPU 高速缓存（Cache L1/L2/L3）的局部性特征。

### 3. DIRECT 直连模式 (`ExecutionMode.DIRECT`)
* **设计定位**：用于执行非常微小的、无锁且不产生任何阻塞的操作，例如简单的入参格式验证、符号转换或最终响应对象的快速拼装。
* **运行特征**：**不进行线程分配与上下文切换**。该任务直接在当前调用者线程（或触发依赖就绪的最后一个工作线程）上同步串行执行。
* **核心目标**：彻底消除线程间传递任务的排队成本及上下文切换开销，使微小步骤的运行效率逼近原生串行调用。

## 容量评估数学模型：利特尔法则 (Little's Law)

为了科学合理地计算线程池的最佳容量，Agelina 引入了基于队列理论和利特尔法则的数学计算模型。

设 $T$ 为目标线程池的最佳工作线程数量：

$$T = N_c \times U_t \times \left(1 + \frac{W_t}{C_t}\right)$$

其中各个变量的定义如下：
* $N_c$：当前 JVM 运行容器所分配的物理 CPU 核心数（Core Count）。
* $U_t$：期望的 CPU 目标利用率（Target CPU Utilization），其值范围在 $0 < U_t \le 1$ 之间。在生产实践中，为保证 GC 回收及系统监控程序留有安全冗余，建议一般设定为 $U_t = 0.8$（即 80% 负载）。
* $W_t$：单个任务在执行周期内的**平均阻塞等待时间**（Waiting Time），例如等待网络 Socket 返回、慢 I/O 读取或等待远程接口响应。
* $C_t$：单个任务在执行周期内的**平均纯 CPU 计算耗时**（Computation Time），即 CPU 处于活跃计算状态的持续时长。

### 比率因子 $W_t / C_t$ 深度分析

等待时间与计算时间的比值（$W_t / C_t$）代表了该类任务的“阻塞系数”：

* **计算密集型任务 (CPU-Bound)**：此时 $W_t \approx 0$（几乎无须等待任何外部信号，任务拉起即可直接算完）。
  代入计算公式：
  $$T \approx N_c \times 0.8 \times (1 + 0) \approx N_c \times 0.8$$
  *架构师建议*：CPU 线程池的容量设为 $N_c$ 或 $N_c + 1$ 是在数学上最为完美的。盲目调大线程池数量不仅无法提升系统的吞吐率，反而会因操作系统疯狂切换线程上下文而导致吞吐暴跌。

* **I/O 密集型任务 (I/O-Bound)**：此时 $W_t \gg C_t$。
  假设一个远程 HTTP 接口调用任务总共耗时 100ms，其中 98ms 都是线程挂起等待 Socket 网络数据包返回，只有 2ms 是在本地对返回的 JSON 进行反序列化和业务校验（此时 $W_t = 98\text{ms}, C_t = 2\text{ms}$）。
  代入计算公式：
  $$T = N_c \times 0.8 \times \left(1 + \frac{98}{2}\right) = N_c \times 0.8 \times 50 = 40 \times N_c$$
  *架构师建议*：IO 线程池必须水平大幅扩展。例如在 8 核的物理环境下，配置 320 到 500 个 IO 工作线程能保证系统达到最佳的吞吐能效。

## 生产环境容量配置示例

Agelina 允许开发人员通过 Spring Boot `application.yml` 配置文件对这些物理隔离池的容量进行微调。推荐生产环境参考以下数学临界值进行配置：

```yaml
agelina:
  executor:
    cpu:
      core-size: 8              # 严格等于 Nc（当前分配的物理 CPU 核心数）
      max-size: 8               # 与核心数保持一致，避免频繁上下文切换
      queue-capacity: 1000      # 队列容量限制极小，便于高负荷下快速响应背压
      keep-alive-seconds: 60
    io:
      core-size: 64             # 根据系统的等待比率系数大幅调大核心线程数
      max-size: 256             # 动态弹性扩容上限
      queue-capacity: 5000      # 队列设为中等长度，可平滑缓冲突发流量
      keep-alive-seconds: 30
```

## 架构审查与规约指南

为了维护 Agelina 隔离设计的完整性，团队在日常代码审查（Peer Code Review）中必须强制遵循以下规约：

1. **DIRECT 直连任务禁忌**：声明为 `DIRECT` 的任务中绝对禁止执行任何可能会引起线程阻塞的代码。如果步骤中包含了 HTTP 请求、数据库 JDBC 操作或 Thread.sleep 等阻塞逻辑，必须立即将其执行模式重新定义为 `ExecutionMode.IO`。否则，直连任务会直接窃取调度引擎或调用方的主线程资源，导致图调度算法发生阻塞瘫痪。
2. **IO 线程池内存红线**：禁止无上限地调大 IO 线程池。在 JVM 默认设置下，每一个新创建的 Java 线程都会在堆外分配约 1MB 物理内存作为其私有栈空间（若未指定自定义 `-Xss` 参数）。如果物理内存受限，应当优先考虑缩减任务队列的大小来更早触发背压机制，而不是无休止地创建线程。
