# 背压管控与过载保护机制架构

在高性能分布式系统中，微服务必须具备抵御突发流量洪峰和下游网络长期中断的能力。无限增长的内存队列与无约束的线程创建是导致内存溢出（OOM）崩溃和系统瘫痪的最主要诱因。

本指南将深入剖析 Agelina 的主动式背压限制设计、队列容量设计模型，以及框架为不同物理线程池自动装配的饱和拒绝执行策略背后的架构考虑。

---

## 背压核心：确定任务队列容量

在幼稚的异步多线程架构中，工作队列通常被默认配置为无界队列（如最大容量为 `Integer.MAX_VALUE` 的 `LinkedBlockingQueue`）。当系统遭遇持续高并发流量压迫时：

1. 外部请求的涌入速率远远超过了工作线程池的消耗处理速率。
2. 数百万个任务在内存队列中迅速积压。
3. 堆（Heap）内存急剧膨胀，频繁触发垃圾回收（GC）引起的全局暂停（STW）。
4. JVM 最终因堆空间耗尽而发生内存溢出（OOM）崩溃。

为了防范这一系统灾难，Agelina 强制要求对所有隔离的线程池声明严格的有界任务队列容量限制，这可以通过 `application.yml` 进行动态微调：

```yaml
agelina:
  executor:
    io:
      queue-type: LINKED_BLOCKING # 安全的有界阻塞队列
      queue-capacity: 5000       # 高吞吐场景下的平滑排队缓冲区
    cpu:
      queue-type: LINKED_BLOCKING
      queue-capacity: 1000       # CPU 密集型任务的极短队列，便于快速背压熔断
```

当任务队列积压达到设定的容量上限时，线程池将进入**饱和状态**。此时，将触发线程池的拒绝执行处理器（`RejectedExecutionHandler`）。

## 饱和拒绝策略：IO 线程池与 CPU 线程池的本质区别

当线程池处于饱和状态时，应用应当如何反应？这完全取决于该池所承载任务的资源特性。基于排队理论，Agelina 对 IO 线程池和 CPU 线程池配置了截然相反的拒绝策略：

```
新异步任务提交 -> [任务队列已满]
                      |
        +-------------+-------------+
        |                           |
        v                           v
  [ IO 线程池 ]               [ CPU 线程池 ]
 饱和拒绝策略：              饱和拒绝策略：
  CALLER_RUNS                 ABORT
        |                           |
        v                           v
在当前提交线程中同步执行任务       立即抛出致命异常
(从而被动减慢请求提交速度)       RejectedExecutionException
        |                           |
   (天然背压限流)              (主动丢载减负)
```

### 1. IO 线程池：`RejectionPolicy.CALLER_RUNS`
I/O 密集型任务的性能瓶颈通常在于外部资源的响应网络时延（如等待数据库返回、等待远程 RPC 接口响应）。

* **运作原理**：当 IO 线程池队列爆满时，触发 `CallerRunsPolicy`。此时，负责提交该异步步骤的当前调用者线程（通常是调度引擎的派发线程或上游的 Servlet 容器工作线程）将被“劫持”用来同步执行这个被拒绝的任务。
* **天然背压机制（Natural Backpressure）**：由于提交任务的线程被迫转去同步执行耗时较长的阻塞式 I/O 步骤，它就无法继续向有向图调度循环提交新的任务，也无法继续从网络 Socket 中接收新的请求数据包。
* **防护效果**：过载压力会顺着调用栈自底向上天然传导。如果 Servlet 容器（如 Tomcat, Netty）的线程全部因执行被拒绝的任务而被占用，整个微服务将自动停止接收新的 TCP 连接。流量压力被安全地挡在系统外部，在网关层表现为短暂的高延迟（或触发 503 重试），从而保护了 JVM 不会因任务无限积压而导致 OOM。

### 2. CPU 线程池：`RejectionPolicy.ABORT`
对于 CPU 密集型任务，系统的性能瓶颈是物理处理器的计算周期。此时，CPU 的物理核心通常已经处于 $100\%$ 的满负荷运转状态。

* **运作原理**：当 CPU 线程池队列爆满时，执行器将毫不犹豫地立即抛出 `RejectedExecutionException`。
* **为什么不能使用 CALLER_RUNS？**：如果让提交线程在 CPU 已经极度饱和的情况下继续去同步执行高计算负荷的任务，只会进一步加剧操作系统内核在物理核心上调度不同线程上下文的激烈争抢，引入疯狂的上下文切换开销，导致整个微服务的整体吞吐率加速崩溃。
* **丢载减负效果（Load Shedding）**：通过果断抛出异常拒绝新任务（即**主动丢载**），Agelina 能够瞬间熔断超载请求，从而确保 CPU 队列中已经排队并处理到一半的存量任务能够快速且正确地算完，确保服务“虽有局部受损，但绝不断电瘫痪”。

## 高可用微服务设计规约（架构师必读）

为了在生产环境构建极高弹性的高可用微服务，建议架构团队强制执行以下规约：

### 1. 科学界定任务队列的深度
严禁盲目调大任务队列。过深的队列会严重掩盖系统瓶颈，导致请求在队列中长时间“排队发霉”。例如：如果您的微服务 p99 SLA 响应时间要求是 500ms，而您的 IO 线程池吞吐速度是 10,000 次/秒，那么 5,000 大小的队列已经能提供 500ms 的吞吐缓冲。如果队列设得更长，请求在队列里还没被线程拉起，就已经超出了客户端的 SLA 超时限制，之后的执行完全是在做无用功。

### 2. 必须配合下游熔断机制
虽然 `CALLER_RUNS` 能够完美保护本地 JVM 内存不溢出，但它本质上是一种压力向上传递和向下积压的策略。如果是外部数据库服务已经瘫痪，`CALLER_RUNS` 持续同步执行数据库操作依旧会给下游数据库塞满连接。因此，必须将 Agelina 与下游熔断器（如 Resilience4j）结合使用，在依赖资源完全不可用时快速 fail-fast，阻止背压变成下游的次生灾害。

### 3. 完美融入优雅停机生命周期
在微服务滚动更新、CI/CD 部署发布时，JVM 进程必须能安全退出。Agelina 自动装配的 `GracefulShutdown` 钩子能够深度钩入 Spring 的 Bean 销毁生命周期。当系统收到停机信号时，框架会给队列中积压的存量任务留出最多 30 秒的“排水时间”，等待全部存量流水线执行完毕后才会强行调用 `shutdownNow()`，保障在滚动发布期间零业务订单数据的非正常丢失。
